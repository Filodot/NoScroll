# Ведение прогресса в GitHub

## 1. Источник истины

- GitHub Project `NoScrol MVP` показывает текущее состояние разработки.
- Один work package из `07-implementation-work-packages.md` соответствует одной GitHub Issue.
- Issue хранит цель, границы, зависимости, критерии приёмки и результат проверки.
- Коммиты и pull request обязательно ссылаются на номер Issue.

## 2. Статусы Project

| Статус | Значение |
|---|---|
| Backlog | Пакет зафиксирован, но ещё не готов к запуску. |
| Ready | Все зависимости закрыты; пакет можно выполнить независимо. |
| In progress | Ведётся один активный сеанс реализации. |
| Review | Код готов, проверки пройдены, требуется итоговая сверка. |
| Done | Критерии приёмки выполнены, изменения находятся в основной ветке. |

Одновременно допускается не более одного `In progress` на одного исполнителя. Интеграционный пакет переводится в `Ready` только после завершения всех указанных зависимостей.

## 3. Поля задач

Каждая Issue содержит:

- идентификатор `WP-xx` или `IP-xx`;
- область: `foundation`, `domain`, `android`, `data`, `ui` или `integration`;
- зависимости;
- ожидаемый результат одного сеанса;
- критерии приёмки;
- обязательные проверки;
- ссылки на PR/коммиты и краткий итог после завершения.

## 4. Метки

- `work-package` — независимый пакет реализации;
- `integration` — пакет объединения готовых частей;
- `blocked` — существует внешняя или незакрытая зависимость;
- `needs-decision` — требуется продуктовое или техническое решение;
- `priority:p0`, `priority:p1`, `priority:p2` — порядок выполнения;
- `area:foundation`, `area:domain`, `area:android`, `area:data`, `area:ui` — область владения.

## 5. Ветки, коммиты и pull request

- Ветка: `codex/<issue-id>-<short-name>`, например `codex/wp-01-policy-engine`.
- Коммит: `<issue-id>: <результат>`, например `WP-01: implement policy decision rules`.
- Заголовок PR начинается с идентификатора пакета.
- В описании PR используется `Closes #<issue>` и перечисляются выполненные проверки.
- Один PR не объединяет несколько независимых work package без явной интеграционной причины.

## 6. Правило обновления прогресса

В начале каждого сеанса задача переводится в `In progress`. После реализации добавляется комментарий с результатом, проверками и известными ограничениями. Затем задача переходит в `Review` и после слияния — в `Done`. Если выполнение остановлено, задача возвращается в `Ready` либо получает `blocked` с точной причиной и следующим действием.

## 7. Начальная готовность

- `WP-00` — `Ready` и является первым пакетом разработки.
- `WP-01`–`WP-15` — `Backlog` до фиксации контрактов в `WP-00`.
- `IP-01`–`IP-06` — `Backlog` до завершения перечисленных в документации зависимостей.
